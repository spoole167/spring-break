Tier 1 · Won't Build spring-boot 3.5 → 4.0

@ConfigurationProperties Field Binding Removed

Spring Boot 4.0 no longer binds @ConfigurationProperties to public fields. Only JavaBean setters or constructor binding work now.
Effort: M

What You'll See

$ mvn spring-boot:run
***************************
APPLICATION FAILED TO START
***************************
Description:
Failed to bind properties under 'app.mail' to com.example.config.MailProperties:

    Property: app.mail.host
    Value: "smtp.example.com"
    Reason: No setter found for property 'host' on class
            com.example.config.MailProperties. Public field binding
            is no longer supported. Add a setter method or use
            constructor binding.

What Changed

Spring Boot 4.0 removed support for binding @ConfigurationProperties values to public fields. The property binding mechanism now requires either JavaBean-style setter methods or constructor binding (via @ConstructorBinding or a single constructor). Classes that relied on direct public field access fail at startup.

// ConfigurationProperties class — field binding (broken)
@ConfigurationProperties(prefix = "app.mail")
public class MailProperties {
    public String host;
    public int port;
    public boolean ssl;
}

@ConfigurationProperties(prefix = "app.mail")
public class MailProperties {
    private String host;
    private int port;
    private boolean ssl;

    public String getHost() { return host; }
    public void setHost(String host) { this.host = host; }
    public int getPort() { return port; }
    public void setPort(int port) { this.port = port; }
    public boolean isSsl() { return ssl; }
    public void setSsl(boolean ssl) { this.ssl = ssl; }
}


// Or use constructor binding (immutable, recommended)
@ConfigurationProperties(prefix = "app.mail")
public class MailProperties {
    public String host;
    public int port;
    public boolean ssl;
}

@ConfigurationProperties(prefix = "app.mail")
public record MailProperties(String host, int port, boolean ssl) {
}

Why This Changed

Public field binding bypassed validation, made immutability impossible, and broke encapsulation. Setter-based and constructor-based binding both support @Validated constraints and are consistent with the rest of Spring's binding model.

How to Fix

A
Add getter/setter methods. Convert public fields to private fields with standard JavaBean getter and setter methods. Your IDE can generate these automatically.
B
Use a Java record (recommended). Convert the class to a record. Records use constructor binding automatically and are immutable by design. This is the cleanest option for configuration properties.
C
Use Lombok @Data. If you prefer fewer boilerplate lines, annotate the class with @Data or @Getter @Setter from Lombok.
✓ Verify: Config props bind correctly — no BindException on startup
Scope check: Search for @ConfigurationProperties classes and check if any have public fields without setters. Look for the pattern public String, public int, etc. in those classes.

Watch Out

This is a startup failure, not a compile error. Your code will compile fine — it fails when Spring Boot tries to bind properties at runtime. If you have good test coverage with @SpringBootTest, you'll catch it in tests. Otherwise it hits production.

If you switch to records, remember that records are immutable. Code that previously mutated the properties object after injection will break. That code was probably wrong anyway, but review it.
