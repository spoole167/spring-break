Spring Batch 6 Listener Support Classes Removed (Runtime)

JobExecutionListenerSupport, StepExecutionListenerSupport, and ChunkListenerSupport are removed in Spring Batch 6.0. Dynamic loading via reflection throws ClassNotFoundException on Boot 4.0.
Effort: S

What You'll See

// Dynamic class loading — e.g. framework, XML config, or legacy reflection
Class.forName("org.springframework.batch.core.listener.JobExecutionListenerSupport");

// Boot 3.5 (Batch 5.x): class present, deprecated since 5.0 — loads fine.

// Boot 4.0 (Batch 6.0): class removed.
java.lang.ClassNotFoundException:
  org.springframework.batch.core.listener.JobExecutionListenerSupport

What Changed

Spring Batch 6.0 (shipped with Spring Boot 4.0) removed three convenience base classes that had been deprecated since Batch 5.0: JobExecutionListenerSupport, StepExecutionListenerSupport, and ChunkListenerSupport. These classes provided empty default implementations of listener methods, a pattern made redundant by Java 8 interface default methods.

// Replace base class with direct interface implementation
public class MyJobListener extends JobExecutionListenerSupport {
    @Override
    public void afterJob(JobExecution jobExecution) {
        // only override what you need
    }
}

public class MyJobListener implements JobExecutionListener {
    @Override
    public void afterJob(JobExecution jobExecution) {
        // implement the interface directly
    }
}

Why This Changed

Java 8 interface default methods eliminate the need for adapter-style base classes. Batch 5.0 deprecated them with that rationale; Batch 6.0 follows through with removal. Implementing the listener interface directly is both more explicit and compatible with composition rather than inheritance.

How to Fix

A
Implement the listener interface directly. Replace extends JobExecutionListenerSupport with implements JobExecutionListener. Do the same for StepExecutionListenerSupport → StepExecutionListener and ChunkListenerSupport → ChunkListener. You only need to implement the methods you actually use — all others have default no-op implementations on the interface.
✓ Verify: JobExecutionListenerSupport, StepExecutionListenerSupport, and ChunkListenerSupport are on the classpath and loadable at runtime
Scope check: Search for extends JobExecutionListenerSupport, extends StepExecutionListenerSupport, and extends ChunkListenerSupport in your source. Also check Spring XML configuration files and any reflection-based lookups for these class names.

Watch Out

The compile-time version of this break is covered by the batch-listener-classes module. This module specifically targets runtime reflection scenarios — XML-configured listeners, framework-generated proxies, or dynamic class loading that bypasses the compiler.
