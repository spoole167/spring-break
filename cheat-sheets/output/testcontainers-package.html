Testcontainers 2.0 Package Relocation

Testcontainers 2.0 moved from org.testcontainers to com.testcontainers. Every import and Maven coordinate changed.
Effort: S ⚙ OpenRewrite

What You'll See

$ mvn compile

  error: package org.testcontainers.containers does not exist
[ERROR] /src/test/java/com/example/IntegrationTest.java:[5,29]

What Changed

Testcontainers 2.0 relocated all packages from org.testcontainers to com.testcontainers. The Maven group ID changed from org.testcontainers to com.testcontainers. Spring Boot 4.0's managed dependencies point to Testcontainers 2.0, so the old coordinates no longer resolve.

// pom.xml — BOM import
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>testcontainers-bom</artifactId>
    <version>1.19.8</version>
    <type>pom</type>
    <scope>import</scope>
</dependency>

<!-- Managed by Spring Boot 4.0 parent POM — remove explicit BOM if using starter-parent -->


// pom.xml — module dependency
<dependency>
    <groupId>org.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>

<dependency>
    <groupId>com.testcontainers</groupId>
    <artifactId>postgresql</artifactId>
    <scope>test</scope>
</dependency>


// Java imports
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

import com.testcontainers.containers.PostgreSQLContainer;
import com.testcontainers.junit.jupiter.Container;
import com.testcontainers.junit.jupiter.Testcontainers;

Why This Changed

The Testcontainers project moved to a commercial stewardship model under AtomicJar / Docker. The package and group ID change reflects the new organisational identity and gives a clean break for the 2.0 API changes.

How to Fix

A
Update group IDs and imports. Change the Maven group ID from org.testcontainers to com.testcontainers in your POM. Then update all Java imports from org.testcontainers to com.testcontainers. See the Testcontainers 2.0 migration guide.
B
Remove explicit BOM if using Spring Boot parent. Spring Boot 4.0's parent POM manages the Testcontainers version. If you previously imported the Testcontainers BOM explicitly, remove it to avoid version conflicts.
✓ Verify: mvn test — Testcontainers start and tests pass
Scope check: Search for org.testcontainers in all POM files, Gradle files, and Java/Kotlin test sources. Every hit needs the package prefix updated.

Watch Out

Spring Boot's @ServiceConnection annotation and DynamicPropertySource patterns still work — but they reference the new Testcontainers 2.0 classes internally. If you have custom @DynamicPropertySource methods that reference container types, those imports need updating too.

If you use Testcontainers in a shared test library or test fixtures module, update that module first. Otherwise downstream projects will get classpath conflicts between the 1.x and 2.x packages.
