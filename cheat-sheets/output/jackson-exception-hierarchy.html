JacksonException No Longer Extends IOException

Jackson 3.0 changed the exception hierarchy so JacksonException extends RuntimeException instead of IOException. Existing catch blocks silently miss Jackson errors, and throws clauses become lies.

What Changed

In Jackson 2.x, JacksonException extended java.io.IOException, so any catch (IOException e) block would also catch malformed-JSON errors. In Jackson 3.0, JacksonException extends RuntimeException. Catch blocks for IOException no longer intercept Jackson parsing failures, and methods that declared throws IOException for Jackson work now throw unchecked exceptions instead.

// Controller error handling — before
try {
    Order order = objectMapper.readValue(body, Order.class);
} catch (IOException e) {
    return ResponseEntity.badRequest().body("Invalid JSON");
}

try {
    Order order = objectMapper.readValue(body, Order.class);
} catch (JacksonException e) {
    return ResponseEntity.badRequest().body("Invalid JSON");
}


// Spring @ExceptionHandler — before
@ExceptionHandler(IOException.class)
public ResponseEntity<String> handleJsonError(IOException ex) {

@ExceptionHandler(JacksonException.class)
public ResponseEntity<String> handleJsonError(JacksonException ex) {

Why This Changed

Jackson parsing errors are not truly I/O errors. The original hierarchy was a design mistake that forced callers to handle checked exceptions even for in-memory parsing. Making JacksonException unchecked aligns with modern Java conventions and removes boilerplate try/catch from code that parses strings or byte arrays.

How to Fix

A
Search for catch blocks. Grep for catch (IOException and catch (JsonProcessingException. Replace with catch (JacksonException where the intent is to handle Jackson parse failures. Keep IOException catches for genuine I/O work (file reads, sockets).
B
Update @ExceptionHandler advice. If your @RestControllerAdvice catches IOException to return 400 on bad JSON, add a separate handler for JacksonException. Otherwise those errors now surface as unhandled 500s.
C
Clean up throws clauses. Methods that declared throws IOException only for Jackson can drop the clause entirely. The compiler won't complain, but the stale declaration misleads readers.
✓ Verify: mvn compile — no JsonMappingException symbol errors
Scope check: Search for catch (IOException, catch (JsonProcessingException, and throws IOException near any ObjectMapper usage. Every hit where the catch was meant to intercept Jackson errors is now a silent miss at runtime. REST controllers, message listeners, and integration test assertions are the most common locations.

Watch Out

This is a silent behaviour change. The code compiles cleanly because JacksonException is now unchecked. You only discover the problem when bad input arrives and the catch block no longer fires, producing 500 errors instead of 400s.

If you catch Exception broadly, you are technically still catching Jackson errors, but you lose the ability to return a domain-appropriate error message. Narrow your catch clauses.

Spring's built-in HttpMessageNotReadableException still wraps Jackson errors for @RequestBody parsing. The risk is in manual ObjectMapper calls inside your own code.
