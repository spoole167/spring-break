HttpClient setSocketTimeout Removed

RequestConfig.Builder.setSocketTimeout(int) is gone in HttpClient 5. Use setResponseTimeout(Timeout) instead.
Effort: S

What You'll See

$ mvn compile
[ERROR] /src/main/java/com/example/HttpComponentsUsage.java:[12,18]
  error: cannot find symbol
    symbol:   method setSocketTimeout(int)
    location: class RequestConfig.Builder

What Changed

Apache HttpClient 5.x removed the integer-millisecond timeout methods (setSocketTimeout(int), setConnectTimeout(int), setConnectionRequestTimeout(int)) and replaced them with overloads that take a strongly-typed org.apache.hc.core5.util.Timeout object.

// Timeout configuration
RequestConfig.custom()
    .setSocketTimeout(5000)
    .setConnectTimeout(3000)
    .build();

import org.apache.hc.core5.util.Timeout;

RequestConfig.custom()
    .setResponseTimeout(Timeout.ofMilliseconds(5000))
    .setConnectTimeout(Timeout.ofMilliseconds(3000))
    .build();

Why This Changed

The integer-millisecond API was ambiguous about units and lacked null-safety. Timeout is explicit about the time unit, supports nanosecond precision, and can represent "no timeout" cleanly.

How to Fix

A
Replace int timeouts with Timeout objects. Import org.apache.hc.core5.util.Timeout and replace each integer-millisecond call with its Timeout.ofMilliseconds(n) equivalent. Also note that setSocketTimeout is now called setResponseTimeout.
✓ Verify: mvn compile — no cannot find symbol for setSocketTimeout
Scope check: Search for setSocketTimeout, setConnectTimeout, and setConnectionRequestTimeout taking integer arguments across all Java/Kotlin sources. Custom HTTP client configuration classes and RestTemplate factory beans are the most common locations.

Watch Out

setSocketTimeout maps to setResponseTimeout in HttpClient 5 — the name changed, not just the type. A search-and-replace on the method name alone will miss this.
