Tier 1 · Won't Build spring-boot 3.5 → 4.0

Jackson 3.0 Group ID & Class Renames

Jackson's Maven coordinates and core class names changed. Every file that imports Jackson is now wrong.
Effort: L ⚙ OpenRewrite

What You'll See

$ mvn compile
[ERROR] /src/main/java/com/example/MySerializer.java:[3,39]
  package com.fasterxml.jackson.databind does not exist
[ERROR] /src/main/java/com/example/MySerializer.java:[12,20]
  cannot find symbol
    symbol: class JsonSerializer

What Changed

Jackson 3.0 moved from com.fasterxml.jackson to tools.jackson. The group ID, packages, and several core classes all renamed in a single release.

// Maven dependency
com.fasterxml.jackson.core:jackson-databind
tools.jackson.core:jackson-databind

// Imports
import com.fasterxml.jackson.databind.JsonSerializer;
import tools.jackson.databind.ser.ValueSerializer;

import com.fasterxml.jackson.databind.SerializerProvider;
import tools.jackson.databind.SerializationContext;

import com.fasterxml.jackson.databind.JsonMappingException;
import tools.jackson.databind.DatabindException;

Why This Changed

Jackson's maintainers wanted to separate the project identity from the original FasterXML organisation. The rename also gave them a clean break to drop deprecated APIs and simplify the class hierarchy.

How to Fix

A
OpenRewrite (recommended). Run the Jackson 3 migration recipe. Handles group ID, imports, and class renames across the entire codebase in one pass.
B
Manual find-and-replace. Update the Maven coordinates first, then fix imports file by file. You need both steps: the group ID change alone won't resolve the class renames.
✓ Verify: mvn compile succeeds with no com.fasterxml errors
Scope check: Search your codebase for com.fasterxml.jackson. Every hit is an import that needs updating. Custom serialisers, deserialisers, ObjectMapper configuration, and DTO annotations are all affected. If you have 5 files that touch Jackson, expect 5 files to change. If you have 200, expect 200.

Watch Out

The class renames aren't just package moves. JsonSerializer became ValueSerializer, SerializerProvider became SerializationContext. A naive search-and-replace of the package prefix will still leave broken class references. Fix both in one pass.

Jackson exceptions (JsonMappingException, JsonProcessingException) are Serializable. If you've persisted these anywhere (dead letter queues, error logs), those serialised streams contain the old class names. That's a separate Tier 2 problem.
