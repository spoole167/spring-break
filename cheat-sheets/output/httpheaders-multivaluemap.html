HttpHeaders No Longer Implements MultiValueMap

HttpHeaders no longer implements MultiValueMap. Code that treats headers as a general-purpose map fails to compile.
Effort: M

What You'll See

$ mvn compile
[ERROR] /src/main/java/com/example/HttpHeadersUsage.java:[8,24]
  error: incompatible types: HttpHeaders cannot be converted to MultiValueMap
[ERROR] /src/main/java/com/example/HttpHeadersUsage.java:[12,16]
  error: cannot find symbol
    symbol: method containsKey(String)

What Changed

HttpHeaders no longer implements MultiValueMap<String, String> or Map<String, List<String>>. Map-style methods like containsKey(), keySet(), and entrySet() are gone from the type. Header-specific methods replace them.

// Replace Map method calls with HttpHeaders equivalents
headers.containsKey("Content-Type")
headers.containsHeader("Content-Type")

Set<String> names = headers.keySet();
Set<String> names = headers.headerNames();

Set<Map.Entry<String, List<String>>> entries = headers.entrySet();
Set<Map.Entry<String, List<String>>> entries = headers.headerSet();

// Passing HttpHeaders where MultiValueMap is expected
void process(MultiValueMap<String, String> map) { ... }
process(headers);
void process(HttpHeaders headers) { ... }
// or, if the method signature can't change:
process(headers.toMultiValueMap());

Why This Changed

Treating HTTP headers as a generic map encouraged misuse and prevented performance optimisations in the header handling implementation. The dedicated HttpHeaders API is more expressive and allows the backing store to be optimised independently of the Map contract.

How to Fix

A
Replace Map methods with HttpHeaders methods. containsKey(k) → containsHeader(k), keySet() → headerNames(), entrySet() → headerSet(). These replacements compile on both 3.5 and 4.0.
B
For method signatures that accept MultiValueMap. If you have a utility method that accepts MultiValueMap<String, String> and you pass HttpHeaders into it, either change the parameter type to HttpHeaders, or call headers.toMultiValueMap() at the call site.
✓ Verify: mvn compile — no incompatible types or cannot find symbol errors on HttpHeaders usage
Scope check: Search for MultiValueMap usages near HttpHeaders across your codebase. Also search for .containsKey(, .keySet(), .entrySet() called on variables of type HttpHeaders.

Watch Out

Code in Spring interceptors, filters, and custom converters that inspects headers using Map idioms is the most common source of these failures. Check HandlerInterceptor, ClientHttpRequestInterceptor, and ResponseBodyAdvice implementations.

toMultiValueMap() returns a copy, not a live view. Mutations to the returned map do not affect the original HttpHeaders.

Further Info
spring-break module: httpheaders-multivaluemap· Spring Framework 7.0 Release Notes
Driven by Spring Framework 7.0. HttpHeaders was a subtype of MultiValueMap since early in Spring's history. The break affects any code that relies on that structural relationship: method parameters typed as MultiValueMap that receive HttpHeaders, or direct calls to Map methods like containsKey(), keySet(), entrySet() on an HttpHeaders reference.
