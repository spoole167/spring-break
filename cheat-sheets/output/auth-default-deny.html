Authorisation Default: Implicit Access → Explicit Deny

Spring Security 7 denies access to all endpoints not explicitly listed in your security configuration. Endpoints that worked via implicit permit now silently return 403.
Effort: M

What You'll See

// SecurityFilterChain — unchanged from Boot 3.5
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/public/**").permitAll()
);
// Note: no .anyRequest() rule

// Before (Spring Boot 3.5) — unlisted paths implicitly allowed
GET /api/health → 200 OK
GET /api/users/me → 200 OK

// After (Spring Boot 4.0) — unlisted paths denied
GET /api/health → 403 Forbidden
GET /api/users/me → 403 Forbidden

// Integration test failures
Expected status: 200
  Actual status: 403

What Changed

Spring Security 7 changed the default policy for endpoints not covered by any requestMatchers rule. Previously, unconfigured paths were implicitly accessible (depending on your filter chain setup). Now, any request that doesn't match an explicit rule is denied with 403 Forbidden. This is an "explicit deny by default" model.

// Add explicit anyRequest() rule
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/public/**").permitAll()
);

http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers("/api/public/**").permitAll()
    .anyRequest().authenticated()  // explicit catch-all
);


// Or if you had truly open endpoints
// /api/health was implicitly accessible
.requestMatchers("/api/health", "/api/info").permitAll()

// Static resources
// /css/**, /js/** implicitly served
.requestMatchers("/css/**", "/js/**", "/favicon.ico").permitAll()

Why This Changed

Implicit access was a security risk — developers would add new endpoints and forget to add corresponding security rules, accidentally exposing them. Denying by default is the safer posture: every endpoint must be explicitly allowed, so new endpoints are locked down until deliberately opened.

How to Fix

A
Add an explicit .anyRequest() rule (recommended). End every authorizeHttpRequests chain with .anyRequest().authenticated() or .anyRequest().denyAll(). This makes your policy complete and explicit. Then add permitAll() rules for truly public endpoints.
B
Audit all endpoints. List every @RequestMapping / @GetMapping / @PostMapping in your app and verify each has a matching security rule. Use Spring Boot Actuator's /mappings endpoint to get the full list.
✓ Verify: All endpoints return 403 unless explicitly permitted in security config
Scope check: Check every SecurityFilterChain bean in your app. If any authorizeHttpRequests block is missing an anyRequest() catch-all rule, all unlisted endpoints are now silently returning 403. Also check for actuator endpoints, static resources, and health checks.

Watch Out

Actuator endpoints like /actuator/health and /actuator/info may be silently blocked. If your load balancer health checks use these, your app will look unhealthy immediately after deployment.

This interacts with CORS preflight requests. OPTIONS requests to unlisted paths now get 403, which browsers interpret as a CORS failure. You may see "CORS error" in the browser console when the real issue is the missing security rule.
