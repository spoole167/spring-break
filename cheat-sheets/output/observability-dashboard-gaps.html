Tier 3 · Wrong Results spring-boot 3.5 → 4.0

Controller/View Spans Silently Disabled

Spring Boot 4.0 disables automatic controller and view rendering spans by default. Observability dashboards show gaps where traces used to appear — but the app itself works fine.
Effort: M

What You'll See

// Grafana / Jaeger trace — before (Spring Boot 3.5)
── HTTP GET /api/orders ─────────────────────────────
  └── controller: OrderController.getOrders  [12ms]
      └── view: orders/list                  [3ms]
          └── jdbc: SELECT * FROM orders     [8ms]

// Grafana / Jaeger trace — after (Spring Boot 4.0)
── HTTP GET /api/orders ─────────────────────────────
  └── jdbc: SELECT * FROM orders             [8ms]
// controller and view spans MISSING

// Dashboard alert
"No data" for panel "Controller Response Time p99"
// SLO breach: "99th percentile latency unknown"

What Changed

Spring Boot 4.0 disabled the automatic instrumentation for Spring MVC controller method spans and view rendering spans by default. The properties management.observations.http.server.requests.enabled and related spring.mvc.observation settings changed their defaults. The traces still record HTTP and JDBC spans, but the controller-level granularity is gone.

// application.properties — re-enable controller spans
# (default in Boot 3.5: controller spans enabled)
management.tracing.enabled=true
spring.mvc.observation.auto-configuration.enabled=true


// Or enable via ObservationRegistry customiser
# (auto-instrumentation was on by default)
@Bean
public ObservationRegistryCustomizer<ObservationRegistry> serverSpans() {
    return registry -> registry.observationConfig()
        .observationHandler(new DefaultMeterObservationHandler(meterRegistry));
}


// Micrometer tracing config
# (no explicit config needed in Boot 3.5)
management.observations.http.server.requests.enabled=true
management.observations.http.client.requests.enabled=true

Why This Changed

The automatic controller spans added overhead (an extra span per request) and duplicated information already available from the HTTP server span. For high-traffic services, the span volume was significant. The new default reduces overhead and storage costs, with opt-in for teams that need the detail.

How to Fix

A
Re-enable controller observation spans. Add spring.mvc.observation.auto-configuration.enabled=true and management.observations.http.server.requests.enabled=true to your application.properties. This restores the controller-level spans in your traces.
B
Update dashboards to use HTTP spans. If the overhead matters, update your Grafana/Datadog dashboards to use the HTTP server span (http.server.requests) instead of the controller span. This span is still present and includes the handler information as attributes.
✓ Verify: All Grafana/Jaeger panels show data — no 'No data' gaps
Scope check: Check your observability dashboards, SLO definitions, and alerting rules. Any panel or alert that queries for controller-level span names (e.g. controller.*) or view rendering spans will show "No data" after the upgrade. Also check custom SpanExporter or ObservationHandler beans that filter by span name.

Watch Out

This is invisible in application logs and functionality. The app works perfectly. The only symptom is missing data in your observability platform — which you might not notice until an incident requires the traces that are no longer there.

If you have SLO alerting based on controller span latency percentiles, those alerts will fire as "no data" — which some alerting systems treat as "OK" (no data = no violations). You could silently lose SLO monitoring.
