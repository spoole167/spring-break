WebSecurityConfigurerAdapter Removed

WebSecurityConfigurerAdapter was deleted from Spring Security. Any config class extending it won't compile.
Effort: M ⚙ OpenRewrite

What You'll See

$ mvn compile
[ERROR] /src/main/java/com/example/SecurityConfig.java:[6,55]
  error: cannot find symbol
    symbol: class WebSecurityConfigurerAdapter
[ERROR] /src/main/java/com/example/SecurityConfig.java:[10,5]
  error: method does not override or implement a method from a supertype
    @Override
    protected void configure(HttpSecurity http)

What Changed

WebSecurityConfigurerAdapter was deprecated in Spring Security 5.7 and removed in 7.0. Configuration is now done by declaring @Bean methods that return SecurityFilterChain, UserDetailsService, etc. The configure(HttpSecurity) and configure(AuthenticationManagerBuilder) override pattern is gone.

// Security configuration class
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            );
    }

    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        auth.inMemoryAuthentication()
            .withUser("user").password("{noop}pass").roles("USER");
    }
}

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            );
        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        var user = User.withUsername("user")
            .password("{noop}pass").roles("USER").build();
        return new InMemoryUserDetailsManager(user);
    }
}

Why This Changed

The adapter pattern forced all security config into a single class and made it hard to compose multiple filter chains. Bean-based configuration is more modular and works naturally with Spring's dependency injection.

How to Fix

A
Convert to @Bean configuration. Remove the extends WebSecurityConfigurerAdapter. Turn configure(HttpSecurity) into a @Bean method returning SecurityFilterChain. Call http.build() at the end. Move authentication config to separate @Bean methods.
B
Use OpenRewrite. The org.openrewrite.java.spring.security6.RemoveWebSecurityConfigurerAdapter recipe automates the conversion.
✓ Verify: mvn compile — no WebSecurityConfigurerAdapter references
Scope check: Search for extends WebSecurityConfigurerAdapter and import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter. Every hit is a class that needs rewriting.

Watch Out

Don't forget http.build(). The bean method must return a SecurityFilterChain, not void. Missing the return statement is a common mistake that gives a confusing null-pointer error at startup.

If you had multiple configure() overrides in the same adapter, split them into separate beans. Authentication configuration should become a UserDetailsService or AuthenticationManager bean, not part of the filter chain.
